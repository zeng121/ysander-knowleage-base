## 一、Agent 与传统软件的本质差异

### 1.1 行为不可控

传统软件：输入 → 确定性输出。代码路径固定。

Agent：输入 → 模型推理 → 工具调用 → 模型再推理 → **循环** → 输出。每一步都有概率分布。

**同一个输入、同一个 prompt、同一套工具，结果可能完全不同。**

### 1.2 不可控不是 bug，是特性

应对策略不是「消除不可控」，而是**约束风险面**：

- **收敛 toolset**：线上只给必要工具，工具少 = 出错面小 = 行为方差降低
- **输出校验层**：独立于 agent 的一层，校验输出格式/业务规则/SLA，校验不过就重试或降级
- **可逆操作优先**：需要不可逆操作的步骤加人工确认
- **硬止损**：超时和预算上限，agent 失控时自动中断

## 二、测试体系重建

传统测试金字塔（unit → integration → e2e）对 agent 基本失效——你没法 mock 一个"正确答案"。

### 2.1 三层测试体系

#### Level 1：格式校验（今天就能做）

纯代码断言，零成本，不需要 golden set：

```
- 输出能 parse 吗？（JSON 合法？）
- 字段齐全吗？（必填字段存在？）
- 类型对吗？（string 不是 number？）
- 超时了吗？（duration < 阈值？）
- 异常输入被正确拒绝了吗？
```

#### Level 2：行为边界（少量数据即可）

不测 agent 答得好不好，测它是否做了不该做的事：

```
- 输入恶意 prompt 时，agent 是否正确拒绝？
- 工具调用次数是否合理？（不陷入死循环）
- 空输入/超长输入是否优雅降级？
```

#### Level 3：质量 Eval（攒够 Golden Set 再做）

- **Golden Dataset**：收集真实对话中「回答真的好」和「回答真的烂」的样本
- **LLM-as-Judge**：用另一个模型打分（1-5 + 理由），不判断完全匹配，只判断质量
- **A/B 对比**：新旧 prompt/toolset/skill 并行跑，对比结果

### 2.2 测试从第一天就要攒的东西

用户少的时候不投资复杂体系，但要做好地基：

1. **自动盯关键指标**（替代人工盯）
   - agent 响应超时 → 报警
   - tool 调用次数 > 阈值 → 可能死循环 → 拉日志
   - 输出 parse 失败 → 降级
   - 连续 fallback → 切备用模型

2. **攒 Golden Set**
   - 每次遇到「这个回答真好」或「这个回答真烂」→ 存下来
   - 不格式化，不打分，就存着
   - 用户量上来后这是 eval set 的种子

3. **可回溯的 trace**
   - 完整轨迹（每一步推理 + 每一步 tool 调用结果）落地
   - 不做 dashboard，就存文件/数据库
   - 出问题时能定位到哪一步跑偏

## 三、数据策略

### 3.1 两类数据，两个目的

| 数据类型 | 内容 | 分析目的 |
|---------|------|---------|
| **对话内容** | 用户输入 + Agent 输出 | 判断「好不好」— 高频场景、回答质量、用户满意度 |
| **后台日志** | tool 调用链 + 推理过程 + 异常 | 定位「为什么」— 根因分析、稳定性、成本 |

**对话内容是你打分的东西，后台日志是你修 bug 的东西。**

### 3.2 落地顺序

**先对话内容，后后台日志。** 对话内容简单、数据量小、直接对应业务价值。日志量大、需要解析，但没对话内容做基准你分析日志也没方向。

### 3.3 最小埋点方案

每次对话结束 → 落一条结构化记录：

```json
{
  "session_id": "xxx",
  "user_input": "用户最后一条消息",
  "agent_output": "Agent 最终输出",
  "tool_calls_count": 5,
  "duration_ms": 3200,
  "ended_by": "normal | timeout | user_abort | error",
  "timestamp": "2026-05-29T12:00:00Z"
}
```

这几个字段够做：
- 高频场景统计（user_input 聚类）
- 问题会话标记（tool_calls > 阈值 或 ended_by != normal）
- 耗时趋势

### 3.4 分析方式

不靠人看，靠 Agent 看：

- cron job 每天让 Agent 读昨天的对话日志
- 自动标出：异常的、用户中断的、跑偏的
- 人只需要每天早上扫一眼日报

## 四、Agent 框架选型的关键维度

从 Codex 迁移的视角：

| 维度 | 重要性 | 说明 |
|------|--------|------|
| **Model-agnostic** | 🔴 必须 | 不绑单一模型，成本可控 |
| **托管式 agent loop** | 🔴 必须 | 不用手搓上下文管理、工具调用、重试、fallback |
| **API 服务化** | 🔴 必须 | 不是 CLI 工具，是对外 HTTP 服务 |
| **Toolset 可控** | 🟡 重要 | 线上只给必要工具，降低行为方差 |
| **SDK/接入方式** | 🟡 重要 | 最好是 OpenAI SDK 兼容，零迁移成本 |
| **更新频率** | 🟢 次要 | Pin 版本即可，不需要每次都追 |

目前已知方案对比：

| 方案 | Model-Agnostic | 托管 Agent Loop | API 服务化 | 自己部署 |
|------|:-:|:-:|:-:|:-:|
| Codex | ❌ | ✅ | ✅ | ❌ |
| Hermes API Server | ✅ | ✅ | ✅ | ✅ |
| Bedrock AgentCore | ✅ | ❌（只提供运行时容器） | ✅ | ❌ |
| LangGraph / AutoGen / CrewAI | ✅ | ❌（需手搓） | ❌（需自建） | ✅ |
| Dify | ✅ | ✅ | ✅ | ✅ |
| Pi (pi.dev) | ✅ | ✅ | ❌（CLI 工具） | ✅ |

## 五、关于 Hermes 的评估

### 优势
- Model-agnostic，DeepSeek 等便宜模型随便换
- OpenAI SDK 兼容，迁移成本低
- Toolset 精细可控，可以按场景裁剪
- Skill 系统做垂类知识注入
- 开源可控、自部署

### 顾虑
- 更新速度极快（12 天从 v0.14 → v0.15，1302 commits）
- 内置工具太多，需要手动收敛
- 无官方 Python SDK（但 OpenAI SDK 兼容，实际不需要）

### 应对
- Pin 版本，不追更
- Toolset 收敛到业务必须的最小集
- 核心接口（/v1/chat/completions, /api/sessions）是稳定的 OpenAI 兼容面
